Introduction

1-3. What is Node JS

  • Node.js is a JavaScript outside of browser.
  • Node.js is a JavaScript runtime built on Chrome's V8 JavaScript engine.
  • Javascript outside of browser. When JS was created, it was used only on browsers. (it was the purpose of it..)
  • created by Ryan Dahl. he wanted to use JS for other stuffs. download Node.js means downloading JS! it's separated from browser.
  • thanks to Node.js, we have React Native(using JS and ReactJS, we can build Android, iOS applications), Electron(you can build desktop apps ex.VS Code, Slack, Facebook Messenger)
  • revolutionary. it changed everything.

1-4. What is NPM

  • npm is a package manager for the JavaScript programming language.
    • npm.Inc. is a subsidiary of GitHub.
    • npm is the default package manager for the JavaScript runtime environment Node.js.
    • free. already have installed. nodeJS comes with npm.
    • yarn is practically same thing, but we'll focus on npm in this course.
      • package ex: express -> 설치: npm install express
  • npm은 nodeJS와 상호작용을 할 수 있게 해줌. npm allows you to interact with nodeJS. it allows you to download packages and many other stuffs.

  • express 어플리케이션 만들기

  • Babel: JavaScript compiler.
    • 아직 nodeJS가 이해하지 못하는 최신 JS 코드가 있음

      2. use Babel -> nodeJS가 JS를 문제없이 이해하도록 변환해줌

Introduciton to Express

3-0. Your first server

  • express(): creates an express application.
    • const app = express(); -> now app is an express application.
  • to make an application listen. listen has a callback.
  • callback: 서버가 시작될 때 작동되는 함수. 서버에게 어떤 port를 listening할지 얘기해줘야 함. 보통 높은 숫자의 port는 비어있음.
  • server: computer that is always online, and connected to the interent. and always listening.
  • 서버를 시작했다면 localhost를 통해서 접속 가능. 브라우저에 localhost:4000 입력
  • to kill your server: control+c

3-1. GET Requests

  • 현재 localhost:5000의 화면에 Cannot Get /가 떠있음.
  • / : 서버의 root
    • ex. google.com === google.com/
  • GET: in this case, HTTP method.
  • HTTP: how we talk to servers or servers communicate with each other. most solid, oldest, and the first one.
  • browsers request to website, and get you the page.

3-2. GET Requests part two

  • express()랑 app.listen() 사이에 we'll configure our application
  1. how to respond to GET request
    • app.get("/", callback_function);
      • ex. app.get("/", () => console.log("Sombody is trying to go home."); //browser에 표시x. 우리 console에 표시될거임.
      • get만 하고 respond를 안하면 브라우저는 평생 기다릴거임ㅠ
  2. how the HTTP request works
    • request: user asks you sth or sends you sth

3-3. Responses

  • route handler has 2 objects
    1. request(req) : first argument
    2. response(res) : second argument
    • those are given to us by express(상속받음)
  • res.end(); -> end the response. server doesn't send browser anything.
  • res.send("Hello World!"); -> browser화면에 string이 표시됨.
  • response를 안해주면 브라우저는 계속 기다림..

3-4. Recap

  • request, response properties -> expreejs.com 들어가서 확인 고
    • ex. res.cookie() : cookie를 보낼 수 있음.
    • res.end() : 접속종료 시키기(finish the connection)
    • res.send() : html을 보낼 수도 있음. but not recommended..
  • express의 blueprint는 아주 작음.
    1. express()
    2. Application
    3. Request
    4. Response
    5. Router: the way we organize routes. handler로 URL들을 정돈.

3-5. Middlewares part one

  • middleware: software in the middle of request and response.
    • 작업을 다음 함수에게 넘기는 함수. not response
    • next();를 포함하면 middleware, 아니면 그냥 controller
    • cf. next(); 전에 return res.send("blah");가 있어도 다음 함수로 안넘어감ㅠ return aborts the mission
  • 용어 정리?
    • middleware is handler and handler is middleware.
    • handler is controller.
  • next argument: call the next function. 말 그대로 get()내부의 '다음' 함수
    • const gossipMiddleware = (req, res, next) => {
      • console.log("I'm in the middle!");
      • next(); //여기서 next는 handleHome
    • };
    • const handleHome = (req, res) => {
      • return res.end();
      };
    • app.get("/", gossipMiddleware, handleHome);
  • all controllers can be middlewares.

3-6. Middlewares part two

  • app.use(): create global middleware that works on any URL.
    • 순서 중요. get()전에 use()가 위치해야함. top to bottom이기 때문
    • ex. app.use(gossipMiddleware);
    • app.get("/", handleHome);
  • cf. 서버 시작하기: npm run dev
  • middleware가 next()를 호출하지 않으면 다음으로 넘어가지 않음.

Routers

4-0. What are routers?

  • router: allow you to organize controllers and URLs. URL을 깔끔하게 만들기 위해 사용.
    • you can think of it as a "mini-application."
    • best way: to organize routes by domain
      • ex. edit-user, delete-user 대신 user/edit, user/delete (grouping)
      • ex. watch-video, edit-video 대신 video/watch, video/edit

4-1. Making our routers

  • global router: /(home), /join, /login, /search, ..
  • 규칙의 예외사항
    • /join 대신 /user/join이 더 말이 됨 but marketing-wise not cool...
    • 우리는 몇몇 url을 짧게 유지할거임.
  • router 만들기
    • const userRouter = express.Router(); ...(1)
    • const handleEditUser = (req, res) => res.send("Edit User"); ...(4)
    • userRouter.get("/edit", handleEditUser); ...(3)
    • app.use("/users", userRouter); ...(2)
    • => localhost:4000/users/edit 으로 이동하면 Edit User가 뜸.

4-2. Cleaning the code

  • good practice: get your work done, and then spend same amount of time cleaning the code. (read "Clean Code", it's a bible for programmers.)
  • Javascript에서 모든 파일은 하나의 모듈 and they are isolated. they are like bubbles. globalRouter.js에서 express를 또 import해야 함.
  • to import globalRouter to server.js, we need to export first. 꼭 필요한 것만 import, export할거임
    • (globalRouter.js 내부) export default globalRouter;
    • (server.js 내부) import globalRouter from "./routers/globalRouter";
      -> default export이기 때문에 함수 이름을 변경할 수 있음. but usually people keep the same names

4-3. Exports

  • routers랑 controllers는 서로 다른 파일에 쓸거임. controller 내부가 방대해지기도 하고, userRoute.js에서는 route만, userController.js에서는 컨트롤러만 다루는게 맞음.
  • userController를 userRouter로 가져오기.
    • option1: export default join (4-2에서 했던 것처럼)
    • option2: controller앞에 export 쓰기. (여러 개를 export)
      • (globalRouter.js 내부) import {join} from "../controllers/userController";
      • (userController.js 내부) export const join = (req, res) => res.send("Join");
      • -> controller 이름 변경 불가능
  • Javascript에서 변수로 쓸 수 없는 단어들: new, delete, ...

4-5. Architecture recap

  • export default ; 딱 하나만 export 가능

4-6. Planning routes

  • (단축키?) command+"../경로+파일이름"클릭 -> 그 파일로 이동!

4-7. URL parameters part one

  • parameter : URL안에 변수를 넣을 수 있음. ":"+something
    • ex. :id, :potato
  • parameter 확인하기: console.log(req.params);
    • or console.log(req.params.id); <- 파라미터가 :id 일 때
  • 순서 중요! videoRouter.get("/upload", upload); 는 videoRouter.get("/:id", see); 전에 위치해야 함. 안그러면 /video/upload로 접근했을 때, express가 upload를 id 취급해버림.

4-8. URL parameters part two

  • regular expressions(정규식): 문자열로부터 특정 정보를 추출해내는 방식.
    • 모든 프로그래밍 언어에 존재.
    • ex. (nico\w+): nico로 시작하는 모든 단어(word)를 선택.
    • ex. (\d+): 숫자만 선택.
    • ex. videoRouter.get("/:id(\\d+)", see); -> 숫자만 id가 될 수 있음.
      • cf. videoRouter.get("/(\\d+)", see); 도 가능 but 값을 request.params.id로 불러올 수 없음.
  • express routing:
    • ex. abcd, abRANDOMcd 등 다 작동.
      app.get('/ab*cd', function(req, res) {
        res.send('/ab*cd')
      })
    • ex. we can have as many b as we want. abbbbcd등.
      app.get('/ab+cd', function(req, res) {
        res.send('/ab*cd')
      })
    • ex. This route path will match acd and abcd.
      app.get('/ab?cd', function(req, res) {
        res.send('/ab*cd')
      })

Templates

5-0. Returning HTML

  • res.send() 내부에 HTML을 전부 쓸 수는 없음. -> pug를 사용합시다.
      pugjs.org, github.com/pugjs/pug, npmjs.com/package/pug 참조
  • Pug: a high-performance template engine heavily influenced by Haml and implemented with Javascript for Node.js and browsers.

5-1. Configuring Pug

  • Pug: 템플릿을 이용해서 뷰를 만드는 것을 도움.
    1. 깔끔한 HTML을 작성하게 해줌.
    2. HTML에 자바스트립트를 포함할 수 있음. (w/ #{})
    3. 반복x. 한 파일로 모든 템플릿을 업데이트할 수 있음. (w/ include)
  • with app, you can set whatever you want. (app.set(...))
    -> view engine도 설정 가능. Pug가 우리의 view engine이 될거임.
    -> now Express will use Pug to return HTML.
  • view: something that user will see.
    by default, how Express looks for view: process.cwd()+'/views'
  • views 폴더 안에 home.pug 파일을 만들고,
    videoController.js에서 const trending = (req, res) => res.render("home");
  • server.js에서 app.set()으로 view engine을 pug로 설정했음 -> 따로 export, import 필요x
  • 우리는 views폴더를 src 폴더 내부에 위치시킴 -> views configuration을 바꿔줘야 함.
    • 정말 구린 해결법 중 하나는 views폴더를 src 폴더 밖으로 꺼내는 것.. 비추
  • cwd : 서버를 기동하는 파일의 위치에 따라 결정됨. 이 경우 package.json. (server.js가 아님)

5-2. Partials

  • app.set()으로 세팅가능 한 것 중 하나 : x-powered-by
    • Express로 앱을 만들면 헤더에 "X-Powered-By:Express"가 생김. 이걸 가리는 것 가능.
  • views configuration 변경하기: app.set("views", process.cwd()+"/src/views");
  • pug에 자바스크립트 코드 넣기: #{}

      else
        li nothing found
  • convention: each item(단수) in items(복수)
  • 자세한 내용은: pugjs.org/language/iteration.html

5-9. Mixins

  • object을 iteration 하려면 object_name.key_name으로 써야 함
    • ex. each video in videos
        li video.title
  • mixin: 데이터를 받을 수 있는 partial (그냥 partial은 데이터를 받지는 못함 ex.footer)
    • smart partial. pre-made HTML block that receives data.
    • block of data that we don't want to copy-paste, and that includes different data.
    • ex. (video.pug 내부, 함수 선언이랑 약간 비슷..)
      mixin video(info)
        div
          h4=info.tilte
          ul
            li #{info.rating}/5
            li #{info.comments} comments.
            li Posted #{createdAt}
            li #{info.views} views.
    • (home.pug 내부)
      include mixins/video
      (...)
      each video in videos
        +video(video)
